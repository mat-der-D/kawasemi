## 1. コンセプト / UX 思想

### 1.1 基本コンセプト

- **人間ユーザーは一人、アクターは複数**という設計。
- 一人鯖運用に特化（複数の人間ユーザーを最初から想定しない）。
- フルスクラッチ実装（fork ではない）。

### 1.2 アクター設計

- 複数アクターは ActivityPub 上で**独立した存在**として振る舞う。
- 「同一オーナー」という概念は**管理層にのみ存在し、プロトコル層には露出させない**。
  - これは**実装の綺麗さ／レイヤー分離**が目的。**擬似匿名性（アクター間の結びつきの秘匿）は非目標**。
  - 同一ドメインに複数アクターが乗る以上、外部からの相関は自明であり隠さない。**単一ドメイン構成でよい**。
- **ローカル配送：意味論は対称、物理配送は最適化**。
  - 意味論レベルでは、ローカル宛でも ActivityPub の意味論（アクティビティ生成・状態遷移・可視性判定など）を遠隔と同一に扱い、ローカル特権的な「振る舞い」を作らない。
  - 物理配送は最適化してよい。ローカル宛を HTTP ループバック＋署名往復にせず、**プロセス内の関数呼び出しでショートカット**してよい。
  - **実装レベルの明記**：最適化してよいのは「**配送関数の選択**」だけ。**Activity の生成・可視性判定・状態遷移は、ローカル／リモートで完全に共通のコードパス**を通す。分岐させるのは最終的な配送手段（in-process 関数呼び出しか HTTP 送信か）のみとする。
  - **注意（ショートカットの罠）**：プロセス内ショートカットは JSON-LD シリアライズ／署名往復を通らないため、**ローカルでは顕在化せずリモート連合でだけ壊れるバグ**（addressing・JSON-LD 展開・署名対象の差異）を見逃しやすい。対策として、ローカル配送でも**論理的に同一の Activity オブジェクトを生成・検証してから**配送関数を分岐させる。ローカル最適化パスと HTTP 連合パスが同じ結果になることをテストで担保する（→ `mastodon-api-compat.md` 4.5）。
- BOT はアクターごとの API トークンで外部から操作可能。

### 1.3 管理画面

- 全アクターを一つのダッシュボードから俯瞰・操作できる管理画面が**目玉**。
- BOT 管理が楽にできることを重視。
- タイムラインの統合表示 vs 個別表示は設定で切り替え可能（使いながら決める）。

### 1.4 運用要件

- インストールが簡単であること。
- 低スペック VPS 上で動作すること。
- コーディングの知識がなくても設定レベルでカスタマイズできること。

## 2. 機能範囲

### 2.1 実装する機能

**投稿・コンテンツ**
- 投稿
- コンテンツウォーニング（CW）
- 投稿編集
- 引用投稿
- 投稿ピン留め
- リッチテキスト（MFM 等）

**インタラクション**
- カスタム絵文字
- 絵文字リアクション
- 投票（Poll）

**プロフィール**
- プロフィールメタデータ（キー・バリュー型）
- フォロー承認制（連合先のみ。同一サーバー内は不要 → 1.2 / 2.3 参照）

**メディア**
- BlurHash
- フォーカルポイント

**モデレーション・安全性**
- Block 送信（Mastodon 互換）
- セキュアモード（オプション設定）

**発見可能性**
- ハッシュタグ

### 2.2 送信は実装しないが、受信は対応する機能

- **アカウント移転**：自分が移転する（送信側 `Move`）は不要。ただし**フォロー中の相手が移転してきた場合の受信側 `Move` には対応する**（フォローの追従が切れないようにする）。これは「実装しない」対象ではなく、受信処理として**サポートする**。
- **通報**：通報の送信は不要。ただし連合先から飛んでくる `Flag` の**受信には対応する**（受け取って破綻させず、管理画面で確認できるようにする）。保存・表示・自動処理の有無といった具体仕様は実装設計時に詰める。

### 2.3 実装しない機能（完全に不要）

- ディスカバリー（プロフィールディレクトリ）。

### 2.4 同一サーバー内のフォロー承認スキップ

- 「同一サーバー内はフォロー承認不要」は、サーバー層がローカル発 Follow を特別扱いする**管理者の明示的な特権**として定義する（1.2 の「意味論は対称」の上に乗る明示的例外）。

## 3. 連合・API

### 3.1 連合基盤（必須）

ActivityPub 連合を成立させる土台。機能トグルではなく前提であり、独自機能と同じ粒度で設計項目として管理する。

- **HTTP Signatures**：送信時の署名生成、受信時の検証、公開鍵の取得・キャッシュ。`draft-cavage`（HTTP Signatures draft）と RFC 9421 の双方、および **double-knocking**（署名方式のフォールバック交渉）への互換対応を前提とする。ブロック先からの署名は拒否する。
- **WebFinger**：`acct:user@domain` → アクター URL の解決。これが無いと外部からフォローできない。複数アクター分のエンドポイントを提供する。
- **NodeInfo**：ソフト名・バージョン・統計の公開。連合先が疎通可否の判断に使う。一人鯖のため公開統計は最小限に留める。
- **署名鍵管理**：アクターごとに署名鍵ペアを生成・保管し、ローテーション経路を確保する。複数アクター設計（1.2）の一部として扱う。
- **inbox / outbox / shared inbox**：単一ドメインに複数アクターが乗る構成（1.2）のため、shared inbox による配送集約を行う。受信は shared inbox と個別 inbox の両方を受ける。
- **アクター・オブジェクトの GET 提供**：`application/activity+json` でのアクター／投稿／コレクションの配信。セキュアモード時は署名付き GET（authorized fetch）を要求する（4.2 のセキュアモードはここに接続される）。
- **JSON-LD / `@context`**：独自拡張（絵文字リアクション等）の `@context` 定義と、受信時の安全な展開（未知プロパティの扱い）。

### 3.2 Mastodon 互換 API

- Mastodon 互換 API は本プロジェクト最大の工数ブロックのため、**独立ドキュメント `mastodon-api-compat.md` に分離**する。機能項目の網羅と工数見積もりは `mastodon-api-estimate.md` を参照。
- 方針の要点：独自機能以外は既存クライアント（Ivory・Elk・Phanpy 等）で完結することを目標。独自機能のフル体験は独自フロント経由。OAuth 2.0・Streaming・Web Push 等の横断サブシステムが前提になる。
- **管理画面・全アクター統合管理は独自フロントでのみサポートする**（Mastodon API は「1 トークン = 1 アカウント」前提のため、標準クライアントでは複数アクターを統合管理できない）。

### 3.3 Misskey API 互換は初期スコープ外（将来検討）

- 初期は **Mastodon API のみ**に集中する。
- 理由：Misskey は独自 API で、フルスクラッチでの両 API 実装は表面積がほぼ倍増。サードパーティ Misskey クライアントのエコシステムは Mastodon より小さく ROI が低い。
- Misskey ライクの独自機能（MFM・絵文字リアクション等）は**独自フロントエンドで体験を完結**させる。

### 3.4 独自機能の連合インターオペラビリティ

絵文字リアクション・引用投稿は標準 ActivityPub に無く、実装ごとに方言が乱立しているため、**独立した設計項目**として扱う。

- 絵文字リアクション：Misskey `Like`+`_misskey_reaction` / Pleroma・Akkoma `EmojiReact` など。
- 引用：`quoteUrl` / `_misskey_quote` / FEP-e232 / Mastodon の引用 など。
- やること：**対応する方言の一覧化・優先度付け・受信時の正規化／送信時の出し分け**方針の策定。
- 工数リスクが高い箇所なので独立管理する。

## 4. 設定・カスタマイズ

### 4.1 確定設定（Mastodon・Misskey 共通機能に対応）

- インスタンス基本情報（名前・説明・管理者情報）
- 文字数制限
- メディアサイズ上限・保存先
- 連合ブロックリスト
- フォロー承認制のデフォルト
- 公開範囲のデフォルト

### 4.2 選択設定（独自・片方のみの機能）

- セキュアモード
- リッチテキストのサポート有無
- 絵文字リアクションの有効/無効
- 引用投稿の有効/無効

### 4.3 インストール時テンプレート（優先度：高）

インストール時に以下のテンプレートを選択可能：

- **Mastodon ライク**：Mastodon に近い機能セットと UI。
- **Misskey ライク**：Misskey に近い機能セットと UI（MFM・絵文字リアクション等が中核）。

- 位置づけ：テンプレートは「**機能セット + 連動 UI プリセット**」であり、**独自フロントエンド前提**。「Misskey クライアント互換」ではない（3.3 参照）。
- 見た目の詳細設定は別途検討（→ 7. 保留）。

## 5. 技術スタック

| 項目 | 選択 |
|------|------|
| 言語（バックエンド） | Rust |
| Web フレームワーク | axum |
| DB | PostgreSQL |
| 非同期ランタイム | tokio |
| DB クエリ | sqlx |
| 非同期タスク処理 | DB キュー（外部依存なし） |
| フロントエンド | TypeScript + React |
| 全文検索 | PostgreSQL 内で完結（外部エンジン不使用 → 5.1 の方針） |

### 5.1 検索：将来の互換性を損なわない最小設計

- 検索処理を**抽象レイヤー（インターフェース）の背後に置く**。呼び出し側（Mastodon 互換 API など）は実装に依存しない。
- 初期は**最小実装**（標準 PostgreSQL の範囲でできる単純な検索から開始）。
- 後から日本語対応に**差し替え可能なマイグレーション経路**を確保（スキーマ・インデックスを後付けできる形に）。
- API 契約は Mastodon の検索 API の形に留め、特定エンジン前提を表に出さない。
- 既知の制約：標準 PostgreSQL の `to_tsvector` は日本語を分かち書きできない。本格的な日本語全文検索には `pg_bigm` 等の拡張が必要だが、これは「入れれば日本語検索が強化される**任意オプション**」に留め、必須にしない（→ 6.5）。

## 6. インストール／配布形態

要件：**インストールが簡単 / 低スペック VPS で動く / 非エンジニアでも立てられる**。
原則：**ユーザーにコンパイル・ビルドをさせない**（配布物は事前ビルド済み）。

### 6.1 基本アーキテクチャ：配布の容易さを優先（単一バイナリは「手段」）

**真の要件は「インストールが簡単」であり、単一バイナリそのものには固執しない**。下記は第一候補だが、容易さを保てるなら他の配布形態（Docker イメージ等）でもよい。

- バックエンドは**事前ビルド済み**（x86_64 / aarch64 Linux 向け）。可能なら musl static の単一バイナリを第一候補とする。
- **React のビルド済み資産をバイナリに埋め込む**（`rust-embed` 等）。Node も別 Web サーバーも不要、バックエンドが SPA を配信。
- **DB マイグレーションを埋め込み、起動時に自動実行**（`sqlx migrate`）。ユーザー操作不要。
- ランタイムは「**アプリ + PostgreSQL**」のみを目指す。
- **メディア処理との両立**：BlurHash・サムネイル・（必要なら）動画処理にネイティブライブラリ（libvips・ffmpeg 等）が要る場合、完全静的単一バイナリは崩れる。その場合は**必要な依存を同梱した Docker イメージ等**で「簡単さ」を担保する方針に切り替えてよい。どこまで pure-Rust で賄い、どこから外部依存を許容するかは別途決める（→ 7. 保留）。

### 6.2 設定の二層化

- **起動設定**（ドメイン・DB 接続先・シークレット）= 小さな **TOML / 環境変数**。
- **運用設定**（インスタンス情報・文字数制限・各種トグル等）= **DB 保存 + 管理画面から編集**。
  - 「コーディング知識なしで設定レベルのカスタマイズ」はこの管理画面で担保する。

### 6.3 PostgreSQL の扱い：両対応

- **外部 PostgreSQL 前提**（OS パッケージ等で用意した Postgres に接続）= 低スペック最優先・主軸。
- **Docker Compose で app + Postgres を同梱**= 「一発」で立てたい人向けの代替。
- → 両方サポート。

### 6.4 HTTPS の扱い：両対応

- **デフォルト = 内蔵 ACME**：バイナリ自身が Let's Encrypt と ACME 交渉し、証明書の取得・自動更新・TLS 終端まで行う（`rustls-acme` 等）。ユーザーは DNS を向けて 80/443 を開けるだけ。nginx も certbot も設定ファイルも不要。
- **オプション = リバースプロキシ後段**：nginx/Caddy 等の後ろに置く構成。アプリは平文で待ち受け、`X-Forwarded-*` を尊重する。既存プロキシ運用者向け。
- **注意（特権ポート bind）**：内蔵 ACME 構成では 80/443 を待ち受けるが、Linux では 1024 未満の特権ポート bind に root か `CAP_NET_BIND_SERVICE` の付与が要る。非エンジニアの躓きポイントになり得るため、**`AmbientCapabilities=CAP_NET_BIND_SERVICE` を設定済みの systemd unit テンプレートを同梱**する等で、ユーザーが権限設定を意識しなくて済むようにする。
- → 両方サポート（主役は内蔵 ACME）。

### 6.5 検索拡張との両立

- 単一バイナリ方式ではユーザー側 Postgres に拡張を入れる必要が出て「簡単」と衝突するため、**初期最小検索は拡張不要の範囲**で実装する（5.1）。
- `pg_bigm` 等は任意オプション扱い（必須にしない）。

## 7. 保留・未決定事項

- 複数アクターの振る舞いの詳細（リアクションの意味論など）。
- 見た目のカスタマイズ詳細。
- 検索機能の詳細仕様（対象範囲・権限など。最小設計の方針は 5.1 で確定済み）。
- 受信側 `Move` / `Flag` の**具体的な**扱い（受信対応そのものは決定済み・2.2。保存／表示／自動処理の詳細仕様のみ保留）。
