# Mastodon 互換 API 設計

`fediverse-design.md` 3.2 から分離した独立ドキュメント。Mastodon 互換 API は本プロジェクト**最大の工数ブロック**であり、表面積が広いため単独で管理する。

最終更新: 2026-06-27

---

## 1. 位置づけ・方針

- 独自機能以外は既存クライアント（Ivory・Elk・Phanpy 等）で完結することを目標とする。
- 独自機能のフル体験は独自フロント経由（→ `fediverse-design.md` 4.3 テンプレート）。
- 詳細な機能洗い出しと工数見積もりは `mastodon-api-estimate.md`（別ファイル）に集約する。**見積もりは Opus による自律 TDD 実装を前提とし、単位は人日ではなく AI ウォールクロックの時間／分。実装スループットではなく設計判断とレビューを律速とみなす**（→ `mastodon-api-estimate.md` 0）。

---

## 2. 互換に必要なサブシステム（見落としやすい土台）

単なる REST エンドポイント群ではなく、以下の**横断的サブシステム**が前提になる。ここを見落とすと工数見積もりが大きく狂う。

- **OAuth 2.0 サーバー**：アプリ登録 / 認可コード / トークン発行。認証が要るすべての API の入口。
- **Streaming API（WebSocket）**：タイムライン・通知のリアルタイム配信。多くのクライアントが事実上前提とする。
- **Web Push（VAPID）**：モバイルのバックグラウンド通知。VAPID 鍵管理と購読管理。
- **ページネーション規約**：`Link` ヘッダ + `max_id` / `since_id` / `min_id`。全リスト系で一貫させる必要がある。
- **エラー・レート制限の互換**：エラー JSON 形・`X-RateLimit-*` ヘッダ。
- **メディアアップロード**：非同期処理（`202` 受理 → 処理完了をポーリング）を含む。

---

## 3. 複数アクターと標準クライアントの関係（重要な割り切り）

- Mastodon API は「**1 アクセストークン = 1 アカウント**」を前提に設計されている。
- 複数のローカルアクターを標準クライアントで扱うと、**アクターごとに別サーバーへログインする**のと同じ体験になる。
- したがって本プロジェクトでは次のように割り切る:
  - **全アクター統合のダッシュボード／BOT 統合管理は独自フロントでのみ提供する**（標準クライアントでは実現しない）。
  - 標準クライアントは「**単一アクターを操作するための窓口**」と位置づける。
  - **管理画面は独自フロントのみでサポート**する（標準クライアント経由の管理機能は提供しない）。

## 4. AI 自律 TDD 開発のための指針

本 API は AI に**自律的に TDD で実装させる**前提。表面積が巨大で、かつ「既存実装の実挙動」が事実上の仕様であるため、以下を開発ループのガードレールとする。

### 4.1 仕様の出典と優先順位

- 一次情報は **実装の実挙動**（Mastodon 本体の実レスポンス）。公式ドキュメントは不完全・後追いのことがあるため、**ドキュメント < 実レスポンス**の優先順位で扱う。
- 既存クライアント（Ivory・Elk・Phanpy）が投げる**実リクエストをキャプチャしてフィクスチャ化**し、受け入れ基準にする。

### 4.2 契約（スキーマ）を先に固定する

- エンティティ単位（Status / Account / Notification / Poll …）の **JSON 形をゴールデン／スナップショットテスト**として先に書き、実装をそれに合わせる。
- AI が実装を進めるときの「正解」をスキーマ契約で先に与えることで、出力のドリフトを防ぐ。

### 4.3 垂直スライスで進める

- エンドポイント単位ではなく、**機能単位（投稿する → TL に出る → 通知される）で縦に 1 本通す**。各スライスが TDD の 1 サイクル。
- MVP 境界を明示し、AI がスコープを無限に広げないようガードレールを置く。段階は **read 系 → write 系 → streaming → push** の順。

### 4.4 決定性を強制する

- 時刻・ID・乱数・署名鍵は**注入可能（DI）**にする。非決定的テストは自律ループを壊す（flaky なテストを見て AI が誤った修正を始める）。
- clock / id generator / RNG を差し替え可能な境界に置く。

### 4.5 テストの階層と責務

- **単体**：ハンドラ・シリアライザ。
- **契約**：エンティティ JSON スキーマ（4.2）。
- **統合**：DB 込みのエンドポイント往復。
- **連合**：自前インスタンスを **2 つ起動**して Activity 往復を検証する。`fediverse-design.md` 1.2 の「意味論は対称」をこの層で担保し、**ローカル最適化パス（in-process）と HTTP 連合パスが同じ結果になることを必ずテストする**。
- **E2E**：実クライアントのキャプチャ再生を最終受け入れ基準にする。

### 4.6 連合の外部依存をテスト可能にする

- 外部実装（Mastodon・Misskey・Pleroma）の**実レスポンスをフィクスチャ化**して契約テストにする。
- ネットワーク・署名検証・公開鍵取得は**モック可能な境界**として切り出す。

### 4.7 自律ループに必要な可観測性

- 失敗時に**リクエスト／レスポンス／実行 SQL**を十分に出力する。AI が原因を特定して自己修正できる診断情報が無いと、ループが停滞または誤修正に陥る。

### 4.8 マイグレーションもテスト対象

- 起動時自動マイグレーション（`fediverse-design.md` 6.1）は up と**データ保持**をテストする。失敗時の安全停止も検証する。
